Foreign Keys & Referential Integrity

// Keeping related tables honest — a row can't point at something that doesn't exist

Concept
Speed
Result0 rows
// Foreign Keys Quick Reference
FOREIGN KEY (col) REFERENCES t(col)
Declares a column must match a value in another table
FOREIGN KEY (customer_id) REFERENCES customers(id)
ON DELETE CASCADE
Deletes child rows automatically when the parent is deleted
Deleting a customer removes their orders too
ON DELETE SET NULL
Child's FK column becomes NULL instead of being deleted
Requires the FK column to be nullable
ON DELETE RESTRICT / NO ACTION
Blocks the delete entirely while children exist
MySQL's default behavior
Referential integrity
Guarantees a FK value always points to a real row (or NULL)
Prevents "orphaned" child records
Requires InnoDB
MyISAM tables ignore FK constraints silently
Always use InnoDB when integrity matters

UNION

Concept
Speed
Reference0 rows
// Normalization Quick Reference
1NF
Every column holds one value, no repeating groups
No product1/product2/product3 columns
2NF
Every non-key column depends on the WHOLE key
No column that only depends on part of a composite key
3NF
Non-key columns depend only on the key
No column depending on another non-key column
Why normalize?
Update a fact once, not in N duplicated places
Avoids inconsistent duplicated data
Trade-off
More tables means more JOINs to read data back
Some systems denormalize intentionally for read speed
Foreign keys
Normalized tables are linked back together
See the Foreign Keys topic

Deadlock

Prevention
Access rows in the same order everywhere

Keep transactions short

MVCC
Keeps multiple row versions via the undo log
Readers get a consistent snapshot, no locks needed
REPEATABLE READ
MySQL’s default isolation level
