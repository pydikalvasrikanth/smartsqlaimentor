Foreign Keys & Referential Integrity

// Keeping related tables honest — a row can't point at something that doesn't exist

Concept
Speed
Result0 rows
// Foreign Keys Quick Reference
FOREIGN KEY (col) REFERENCES t(col)
Declares a column must match a value in another table
FOREIGN KEY (customer_id) REFERENCES customers(id)
ON DELETE CASCADE
Deletes child rows automatically when the parent is deleted
Deleting a customer removes their orders too
ON DELETE SET NULL
Child's FK column becomes NULL instead of being deleted
Requires the FK column to be nullable
ON DELETE RESTRICT / NO ACTION
Blocks the delete entirely while children exist
MySQL's default behavior
Referential integrity
Guarantees a FK value always points to a real row (or NULL)
Prevents "orphaned" child records
Requires InnoDB
MyISAM tables ignore FK constraints silently
Always use InnoDB when integrity matters

// UNION Quick Reference
UNION
Combines results, removes exact duplicate rows
Costs extra work: implicit dedupe sort
UNION ALL
Combines results, keeps duplicates
Faster — no dedupe step
Same column count
Every SELECT in the UNION must return the same number of columns
3 columns + 3 columns, not 3 + 2

Column N of each SELECT should share a compatible data type

Views

// A saved query that behaves like a table

Concept
Speed
Result0 rows
// Views Quick Reference
CREATE VIEW v AS SELECT ...
Saves a query under a name; the view stores no data of its own
CREATE VIEW eng_staff AS SELECT * FROM employees WHERE dept='Eng'
SELECT * FROM view_name
Runs the underlying query fresh every time — always current data
Behaves exactly like querying a table
Updatable view
A simple, single-table view (no GROUP BY/JOIN/aggregate) can be inserted/updated through
UPDATE eng_staff SET salary=... writes to employees
Read-only view
Views with JOINs, GROUP BY, DISTINCT, or aggregates cannot be written through
MySQL rejects INSERT/UPDATE on them
Why use views?
Hide complexity, enforce row/column-level security, reuse common joins
Grant SELECT on a view instead of the base table
DROP VIEW v;
Removes the view definition — never touches underlying data
Safe to drop anytime; base tables are untouched

Transactions & ACID

// Grouping statements so they succeed or fail together

Concept
Speed
Transaction Log
// Transactions & ACID Quick Reference
START TRANSACTION / BEGIN
Opens a transaction — changes are staged, not yet permanent
Followed by one or more statements
COMMIT
Makes all staged changes permanent and visible to others
Cannot be undone after this point
ROLLBACK
Discards every change made since the transaction started
As if the transaction never happened
Atomicity
All statements in the transaction succeed, or none do
A transfer debits one account and credits another together
Consistency, Isolation, Durability
Data stays valid; concurrent transactions don't interfere; committed data survives crashes
Together with Atomicity, this is ACID
Isolation levels
READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ (MySQL default), SERIALIZABLE
Trade-off: stricter isolation = more locking, less concurrency
TOPIC 13 — INTERMEDIATE

Basic Normalization

// Structuring tables to eliminate redundancy and update anomalies

Concept
Speed
Notes
// Normalization Quick Reference
1NF
Every column holds a single, atomic value — no comma-separated lists or repeating groups
Split a "phones" column of '555-1,555-2' into separate rows
2NF
1NF, plus every non-key column depends on the whole primary key, not just part of it
Fixes issues in tables with composite keys
3NF
2NF, plus no non-key column depends on another non-key column (no transitive dependency)
Move dept_location out of employees into a departments table
Update anomaly
Redundant data means one fact must be changed in many places, or copies drift out of sync
Changing a department's location in 50 employee rows
Insertion anomaly
You can't record a fact (e.g. a new department) until an unrelated fact exists (an employee in it)
Fixed by giving departments their own table
Trade-off
Normalizing reduces redundancy but adds JOINs; denormalizing is sometimes chosen for read speed
Real schemas often stop at 3NF and denormalize selectively

BEFORE INSERT/UPDATE
Runs before the row is written — can inspect or modify NEW.col values

Write to an audit_log table
NEW / OLD

TOPIC 16 — ADVANCED

Locking

// How MySQL keeps concurrent writers from stepping on each other

Concept
Speed
Connection Timeline
// Locking Quick Reference
Row lock (InnoDB default)
Only the specific rows touched by a statement are locked; other rows stay free
High concurrency for OLTP workloads
Table lock
Locks the entire table; used by MyISAM, or explicitly via LOCK TABLES
Blocks all writers, even to unrelated rows
Shared (S) vs Exclusive (X) lock
Many transactions can hold S locks at once (reads); only one can hold an X lock (writes)
SELECT ... FOR SHARE vs FOR UPDATE
Deadlock
Two transactions each hold a lock the other needs — neither can proceed
T1 locks row A then wants B; T2 locks B then wants A
Deadlock resolution
InnoDB detects the cycle and automatically kills one transaction (rolls it back) to let the other proceed
The killed transaction gets "ERROR 1213: Deadlock found"
Avoiding deadlocks
Access rows in a consistent order across transactions; keep transactions short
Always update accounts in ascending id order

Query Optimization Beyond Indexes

// Reading the optimizer's plan and steering it when needed

Concept
Speed
EXPLAIN Output
// Query Optimization Quick Reference
EXPLAIN SELECT ...
Shows the plan MySQL intends to use, without running the query
EXPLAIN ANALYZE also shows real timing
type: ALL vs ref/range vs const
ALL means a full table scan; ref/range/const mean an index is narrowing the search
Aim to avoid "ALL" on large tables
Nested Loop Join
For each row in the outer table, scan the inner table (or its index) for matches
MySQL's primary join strategy
Hash Join
Builds an in-memory hash table on one side, then probes it — used for equi-joins without a usable index (MySQL 8.0.18+)
Often faster than nested loop for large unindexed joins
STRAIGHT_JOIN
Forces the join order to match the order tables are written in the query
Use when the optimizer picks a bad join order
Index hints: USE/FORCE/IGNORE INDEX
Overrides the optimizer's index choice for a specific table
SELECT ... FROM t FORCE INDEX (idx_name)
TOPIC 18 — ADVANCED

Partitioning

// Splitting one logical table into physically separate chunks

Concept
Speed
Query Scan Plan
// Partitioning Quick Reference
PARTITION BY RANGE(col)
Rows are routed into partitions by which value range col falls into
Common for partitioning by year or date
PARTITION BY HASH(col)
Rows are spread evenly across N partitions using a hash of col
Good for even load distribution, not range queries
Partition pruning
The optimizer skips partitions it can prove contain no matching rows
WHERE order_date >= '2024-01-01' only scans 2024's partition
Why partition?
Faster maintenance (drop old partitions instantly), better I/O locality, parallelizable scans
DROP PARTITION p2020 instantly purges old data
Not a replacement for indexes
Partitioning narrows which chunk to scan; indexes still narrow rows within that chunk
The two techniques work together
Partition key limitation
Every unique key (including the primary key) must include the partitioning column
Can restrict schema design choices

JSON Data Type & Functions

// Storing and querying semi-structured data inside a relational column

Concept
Speed
Result
// JSON Quick Reference
JSON column type
Stores validated JSON in a binary format; invalid JSON is rejected on insert
CREATE TABLE t (attrs JSON)
JSON_EXTRACT(col, path)
Pulls a value out of a JSON document by path
JSON_EXTRACT(attrs, '$.color')
-> and ->> operators
Shorthand for JSON_EXTRACT; ->> also unquotes the result as text
attrs->>'$.color' returns 'red', not '"red"'
JSON_SET(col, path, value)
Returns a new JSON document with one key updated or added
UPDATE t SET attrs = JSON_SET(attrs, '$.color', 'blue')
Generated column + index
A virtual column extracted from JSON can be indexed, since JSON itself can't be indexed directly
ALTER TABLE t ADD color VARCHAR(20) GENERATED ALWAYS AS (attrs->>'$.color')
When to use JSON vs columns
Good for flexible/sparse attributes; regular columns are better when you filter/index/report on a field constantly
Product custom attributes vs core price/name fields

Full-Text Search

// Ranked, relevance-based search across text columns

Concept
Speed
Result
// Full-Text Search Quick Reference
FULLTEXT INDEX (col)
A special index that tokenizes text into searchable words, with relevance scoring
ALTER TABLE articles ADD FULLTEXT(title, body)
MATCH(cols) AGAINST('query')
Natural language mode (default) — ranks rows by relevance score, best matches first
Words below the 50% document frequency stoplist threshold are ignored
AGAINST('query' IN BOOLEAN MODE)
Supports operators for required/excluded/weighted words instead of pure ranking
AGAINST('+mysql -oracle' IN BOOLEAN MODE)
+word
In boolean mode: this word must be present
+mysql requires "mysql" in the row
-word
In boolean mode: this word must NOT be present
-oracle excludes rows mentioning "oracle"
Why not LIKE '%word%'?
LIKE with a leading wildcard can't use any index — full-text search is indexed and ranked
FULLTEXT scales to large text columns; LIKE doesn't
